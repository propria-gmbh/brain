# Чек-боксы `- [ ]` в .md файлах — когда это нарушение, когда нет

Контекст: 2026-06-23 в `08_PROJECTS/` нашлось 8 файлов с чек-листами, дублирующими `tasks.json` (`safe-min/07_task_backlog.md` и др.) — часть пунктов там не существовала в `tasks.json` вообще и могла потеряться при удалении. Это правило фиксирует, как отличать один тип файла от другого, чтобы не путать снова.

## Запрещено (удалять/переносить в tasks.json)

**Таск-бэклог** — список будущих действий, который существует, чтобы спланировать и затем выполнить работу. Признаки:
- Название файла говорит само за себя: `*_backlog.md`, `*_plan_*.md` со списком действий (не эпиков с DoD)
- Пункты — это будущие шаги без привязки к конкретной уже идущей сущности (домен, магазин, письмо)
- Нет дат у `[x]`-пунктов — не журнал, а чистый план

→ Перенести недостающие пункты в `tasks.json` (`tools/add_task.py`), затем удалить чек-лист из .md.

## Разрешено (не трогать)

**1. Эпик-план с Definition of Done** — план разработки конкретного артефакта (сайт, фича), где чек-бокс = шаг эпика, а не отдельная "задача" в смысле tasks.json. Пример: `safe-min/10_website_plan.md`. Отличие от backlog: у каждого эпика есть DoD, чек-боксы — это его внутренняя структура, не параллельный список приоритетов.

**2. Процедурный чек-лист (runbook)** — повторяемая инструкция для операции, которая будет выполняться много раз (проверка домена, настройка магазина). Чек-бокс здесь — шаг процедуры, а не "дело, которое нужно сделать к дате". Обычно лежит в папке `checklists/`. Примеры: `ecom/checklists/domain-verification.md`, `ecom/checklists/andrey-checklist.md`, `ecom/checklists/domain-buying.md`. Эти файлы явно перечислены как "Ключевые файлы" в `CLAUDE.md` — не удалять.

**3. Журнал/лог конкретной сущности** — датированные `[x]`-записи (что и когда сделано/проверено) по одному домену/магазину/поставщику, иногда с парой открытых пунктов "осталось проверить". Это аудит-трейл, не backlog. Примеры: `ecom/domains-hotlist.md`, `ecom/gmc/charlieandted-us-setup.md`, `ecom/supplier-kungfubuy.md`, `ecom/stores/us-new/hypothesis-test.md`. Также короткие исторические `[x] YYYY-MM-DD ...` записи в файлах проектов (`jobcenter.md`, `mitya.md`, `health.md`, `sailing-catamaran-2026.md`).

**4. Ритуальные чек-листы** — `05_PLANS/recurring/*.md` (утренний/недельный/месячный чеклист) и `05_PLANS/today.md` — явно разрешены в `CLAUDE.md`, не относятся к этому правилу.

**5. Шаблоны** — `_templates/*.md` с примером структуры (placeholder-чекбоксы) — не реальные задачи.

## Как проверять перед удалением

1. `grep -c '\- \[ \]' file.md` — найти открытые пункты.
2. Прочитать контекст: это план будущей работы (запрещено) или лог/процедура/эпик (разрешено)?
3. Если backlog — сверить каждый пункт с `tasks.json` (`grep -i` по ключевым словам). Недостающие — добавить через `add_task.py` **до** удаления файла.
4. Не удалять файл целиком, если в нём есть таблицы/контекст, не относящийся к чек-листу (метрики, критерии решений) — выносить чек-лист, контекст оставлять.
