# Предложение: архитектура системы задач

## Проблемы текущей системы

1. **Уровень — это заголовок раздела, а не атрибут задачи**
   Результат: задачи в project-файлах (jobcenter.md, sailing-catamaran-2026.md, supplier-kungfubuy.md) вообще вне системы приоритетов. Нельзя одним запросом получить "все L1 задачи".

2. **Дублирование**: personal.md содержит ссылку на jobcenter.md, а jobcenter.md содержит задачи. Где канонический источник?

3. **Someday — атрибут в голове, но нигде не зафиксирован**
   Концептуально правильно, практически — нет единого формата.

4. **today.md формируется вручную** без опоры на атрибуты → часть L1 задач забывается.

---

## Предлагаемая архитектура

### Принцип 1: задача живёт ровно в одном месте

| Тип задачи | Где хранится |
|---|---|
| Не привязана к проекту | `05_PLANS/tasks/<domain>.md` |
| Привязана к активному проекту | `08_PROJECTS/<project>.md` |
| Новая, не разобранная | `05_PLANS/inbox.md` |

Нет задач в двух местах. personal.md не дублирует jobcenter.md — либо ссылка (текущий вариант), либо задача живёт только в одном из файлов.

---

### Принцип 2: уровень — инлайн-атрибут задачи

Формат: `[L1]`, `[L2]`, `[L3]` в конце строки задачи.

```markdown
- [ ] Сдать Q1 отчётность Propria [L1] ❗
- [ ] Запустить магазин США [L2]
- [ ] Сделать кино про Турцию [L3]
```

Зачем:
- `grep "\[L1\]" **/*.md` — все критичные задачи за 1 секунду
- Задачи в project-файлах получают тот же атрибут
- Уровень виден без навигации по разделам

Заголовки `## Уровень 1 — Фундамент` → убрать или оставить как визуальную группировку, но НЕ как единственный носитель приоритета.

---

### Принцип 3: someday — атрибут `[someday]`

```markdown
- [ ] Сделать медиацентр [L3] [someday]
- [ ] Изучить StiftungsHolding [L2] [someday]
```

Еженедельный ритуал: grep `[someday]` → выбрать 2–3 задачи в активный пул → убрать `[someday]`, поставить в нужный домен.

---

### Принцип 4: дедлайн и ожидание — инлайн

```markdown
- [ ] Нарисовать картины [L3] (дедлайн: 2026-05-22)
- [ ] Страховой агент ARAG [L1] (ждёт: 2026-05-06)
```

---

### Принцип 5: today.md — ручной список из L1

today.md не меняет архитектуру. Это дневной фокус: 3–5 L1 задач, которые делаются сегодня. Остальные L1 — в domain/project файлах.

Альтернатива (если захочется автоматизации): Obsidian Dataview или скрипт для grep всех `[L1]` без `[someday]` → генерация today.md.

---

## Структура после изменений

```
05_PLANS/
  inbox.md          ← захват, разбирать раз в день
  today.md          ← дневной фокус (3–5 L1)
  tasks/
    personal.md     ← личные задачи без проекта, с [L1]/[L2]/[L3]
    ecom.md         ← то же, для ecom домена
    propria.md      ← то же
    finance-personal.md
  recurring/        ← повторяющиеся (daily/weekly/monthly)

08_PROJECTS/
  jobcenter.md      ← задачи [L1], живут здесь, не в personal.md
  sailing-catamaran-2026.md  ← задачи [L2]/[L3]
  ecom/supplier-kungfubuy.md ← задачи [L1] (активный блокер)
  ...
```

---

## Миграция

**Шаг 1** (15 мин): добавить `[L1]`/`[L2]`/`[L3]` к задачам в domain-файлах.
**Шаг 2** (5 мин): добавить атрибуты к задачам в project-файлах (jobcenter, sailing, kungfubuy).
**Шаг 3** (5 мин): убрать дублирование — задача либо в domain, либо в project, не ссылка + копия.
**Шаг 4** (опционально): написать grep-команду или Obsidian Dataview для L1 дашборда.

Заголовки разделов `## Уровень 1` можно оставить как визуальную группировку — они не мешают атрибутам.

## Открытые вопросы для обсуждения

1. Оставить заголовки `## Уровень 1` как визуальную группировку — или убрать совсем?
2. Формат атрибута: `[L1]` / `#L1` / `:L1` — что удобнее в Obsidian?
3. Нужен ли L1-дашборд (автоматический список) — или today.md достаточно?
4. Как быть с project-файлами типа ecom/ (папка со множеством файлов) — задачи в одном tasks.md внутри проекта?
